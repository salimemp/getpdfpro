# Platform setup

The Flutter project shells (`ios/`, `android/`) are not committed to the repo — they're generated by `flutter create` on first setup. The customized configuration files that should live inside those generated directories are tracked here as **source-of-truth templates**.

## Why this approach

We used to commit the generated `ios/Runner/Info.plist` and `android/app/src/main/AndroidManifest.xml` directly, but that caused two problems:

1. **Merge conflicts** every time a teammate ran `flutter create --platforms=ios,android .` with a slightly different CLI version.
2. **Lost changes** when a CI step regenerated the platform shells.

So now we commit the **customizations** (URL scheme registration, intent filters, speech permissions) and let `flutter create` provide the boilerplate. The `apply.sh` script merges them.

## First-time setup (after cloning)

```bash
cd apps/mobile
flutter create --org com.getpdfpro --platforms=ios,android .
flutter pub get
bash platform_setup/apply.sh
cd ios && pod install && cd ..
```

After `apply.sh` runs, your `ios/Runner/Info.plist` and `android/app/src/main/AndroidManifest.xml` contain the deep-link + permission customizations that make OAuth, magic links, password reset, the Dictate tool, and the camera scanner work.

## What's customized

### iOS — `Info.plist`

- **`CFBundleURLTypes`** — registers the `getpdfpro://` custom URL scheme so iOS opens our app for `getpdfpro://login-callback#access_token=...` links. Required for Supabase OAuth + magic-link callbacks.
- **`FlutterDeepLinkingEnabled`** — enables Flutter's built-in deep-link handling for the engine's webview-style navigation. Required for the new `app_links` 6.x behavior on iOS.
- **`NSMicrophoneUsageDescription`** + **`NSSpeechRecognitionUsageDescription`** — speech-to-text permission strings for the Dictate tool and voice-command search. Without these, iOS hard-fails the first time `speech_to_text` is invoked.
- **`NSCameraUsageDescription`** + **`NSPhotoLibraryUsageDescription`** — camera + photos permissions for the Scan-to-PDF and Image-to-PDF tools.

### Android — `AndroidManifest.xml`

- **Custom scheme intent filter** — `<data android:scheme="getpdfpro"/>` so Android opens our app for `getpdfpro://...` links.
- **Universal Links intent filter** — `autoVerify="true"` for `https://app.getpdfpro.com/auth/*` and `/reset-password`. Currently a stub (no `/.well-known/assetlinks.json` yet) so the OS will fall back to the custom-scheme flow; activate after deploying the asset-links file.
- **`<queries>`** — Android 11+ (API 30+) requires apps to declare which external packages they can resolve. We declare HTTPS + HTTP browser intents so the OAuth flow can launch a Custom Tab without hitting "no app found".
- **Permissions** — `INTERNET`, `CAMERA`, `RECORD_AUDIO`, `READ_MEDIA_IMAGES` (API 33+), `READ_EXTERNAL_STORAGE` (API ≤32), `POST_NOTIFICATIONS`. Camera + microphone are declared as **optional** hardware features so the app installs on devices without them (e.g. Android TV, tablets without a mic).

1. Get the SHA-256 fingerprint of the signing key:
   ```bash
   cd android
   ./gradlew signingReport
   ```
2. Deploy `https://app.getpdfpro.com/.well-known/assetlinks.json` with the package name + SHA-256 fingerprint.
3. Wait for Google's crawler to verify (usually minutes to hours).
4. Re-build the Android app — verified App Links open directly without the disambiguation dialog.

## Supabase dashboard allowlist (REQUIRED for password recovery to work)

The custom-scheme deep links (`getpdfpro://login-callback`, `getpdfpro://reset-password`, `getpdfpro://confirm-email`) only work if Supabase's verify endpoint recognizes them as valid redirect targets. Otherwise the email link's `?code=...` is **silently dropped** and the user lands on the Site URL with a JSON error like `{"error": "requested path is invalid"}`.

**One-time setup (per Supabase project):**

1. Open <https://app.supabase.com/project/YOUR_PROJECT/auth/url-configuration>
2. Under **Redirect URLs**, add:
   ```
   getpdfpro://login-callback
   getpdfpro://reset-password
   getpdfpro://confirm-email
   getpdfpro://**
   ```
   The wildcard `getpdfpro://**` covers any future hosts (e.g. `getpdfpro://magic-link`) without needing to come back to this page.
3. Click **Save**.

**How to verify it's working:**

```bash
# On Android
adb shell am start -a android.intent.action.VIEW -d "getpdfpro://reset-password?code=fake"

# On iOS
xcrun simctl openurl booted "getpdfpro://reset-password?code=fake"
```

Both should open the app, and `flutter run`'s debug console should show:
```
AuthStateChange.passwordRecovery — routing to /auth/recovery
```

If you see `{"error": "requested path is invalid"}` somewhere, the allowlist is missing an entry.

**Testing the three flows end-to-end:**

| Flow | Trigger from terminal | Expected behavior |
|------|----------------------|-------------------|
| Password recovery | `xcrun simctl openurl booted "getpdfpro://reset-password?code=fake"` | App opens → /auth/recovery → "verifying..." → error after 8s |
| Email confirmation | `xcrun simctl openurl booted "getpdfpro://confirm-email?code=fake"` | App opens → /auth/confirm → "verifying..." → error after 8s |
| OAuth callback | `xcrun simctl openurl booted "getpdfpro://login-callback#error=test"` | App opens → /login with the OAuth error surfaced |

(Replace `xcrun simctl openurl booted` with `adb shell am start -a android.intent.action.VIEW -d` for Android.)

## Updating the templates

If you add a new permission or intent filter:

1. Edit the file in `platform_setup/ios/Info.plist` or `platform_setup/android/AndroidManifest.xml`.
2. Re-run `bash platform_setup/apply.sh` to copy the changes into the live project.
3. Commit both the template and the regenerated platform file.

## Testing deep links locally

```bash
# iOS Simulator
xcrun simctl openurl booted "getpdfpro://login-callback#access_token=test"

# Android Emulator
adb shell am start -a android.intent.action.VIEW -d "getpdfpro://login-callback#access_token=test"
```

Both should open the app. The router's deep-link listener will parse the URL and route accordingly. `debugPrint` output in `flutter run` will show the typed action received.
